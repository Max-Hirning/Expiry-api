# Chat Module

Real-time messaging system for team collaboration with message editing, deletion, and read status tracking.

## Module Structure

```
src/modules/chat/
├── index.ts                 # Entry point, exports autoPrefix
├── chat.route.ts           # REST API route definitions
├── chat.handler.ts         # Request handlers (thin layer)
├── chat.service.ts         # Business logic and database operations
└── chat.socket-handler.ts  # Socket.IO event handlers for real-time messaging
```

## Service Methods

Internal methods used by other modules (e.g., document module creates chats):

- **`createChat()`** — Create a new chat with initial members
  - Creates a chat record and adds members in transaction
  - Used internally to set up document chats

- **`upsertChatMember()`** — Add members to an existing chat
  - Adds or updates existing members in transaction
  - Updates member status, name, avatar on conflict

- **`updateChat()`** — Update chat settings (name, AI agent toggle, AI message visibility mode)

## API Endpoints

All endpoints are prefixed with `/api/chats/:teamId` and require authentication.

### Chats

- **GET `/:teamId`** — Fetch paginated list of chats for authenticated user
  - Query: `limit` (default 20), `cursor` (for pagination)
  - Response: Chat list with cursor pagination
  - Action: `GET_CHATS`

- **GET `/:teamId/:chatId`** — Fetch single chat with active members
  - Response: Chat with active members
  - Action: `GET_CHAT`

- **PUT `/:teamId/:chatId`** — Update chat settings
  - Body: `{ name?: string, aiAgentEnabled?: boolean, aiAgentVisibility?: "ALL" | "SENDER_ONLY" }` (at least one field required)
  - Response: Updated chat
  - Authorization (`jwt.ts`): the initiator must be a team member with role `OWNER` or `ADMIN`, **and** an active `ChatMember` of the target chat
  - Broadcasts: `chat:updated` to all users in chat room
  - Action: `UPDATE_CHAT`

### Messages

- **GET `/:teamId/:chatId/messages`** — Fetch paginated messages in a chat
  - Query: `limit` (default 20), `cursor`, `parentMessageId` (for threaded messages)
  - Response: Message list with cursor pagination
  - Action: `GET_MESSAGES`

- **POST `/:teamId/:chatId/messages`** — Send a new message
  - Body: `{ message: string, parentMessageId?: string }`
  - Response: Created message
  - Action: `SEND_MESSAGE`

- **PATCH `/:teamId/:chatId/messages/:messageId`** — Edit a message
  - Body: `{ message: string }`
  - Constraint: Only message author can edit, within 5-minute window, cannot edit auto-generated messages
  - Response: Updated message
  - Action: `EDIT_MESSAGE`

- **DELETE `/:teamId/:chatId/messages/:messageId`** — Delete a message
  - Constraint: Only message author can delete, within 5-minute window, cannot delete auto-generated messages
  - Response: Deleted message
  - Action: `DELETE_MESSAGE`

- **POST `/:teamId/:chatId/messages/read`** — Mark messages as read
  - Body: `{ messageIds: string[] }`
  - Response: Created read status records
  - Action: `MARK_MESSAGES_READ`

## Socket.IO Events

Established after authentication. User data is attached to socket via middleware.

### Client → Server

- **`chat:join`** — Join a chat room
  - Payload: `{ chatId: string, teamId: string }`
  - Validation: User must be active chat member
  - Response: `chat:joined` event

- **`chat:leave`** — Leave a chat room
  - Payload: `{ chatId: string }`

- **`chat:message:send`** — Send message via Socket.IO (alternative to REST)
  - Payload: `{ chatId: string, teamId: string, message: string, parentMessageId?: string }`
  - Broadcasts: `chat:message:new` to all users in chat room

- **`chat:messages:read`** — Mark messages as read via Socket.IO
  - Payload: `{ chatId: string, teamId: string, messageIds: string[] }`
  - Broadcasts: `chat:messages:read` to all users in chat room

### Server → Client (Broadcasts)

- **`chat:message:new`** — New message sent
  - Payload: Message object with author details

- **`chat:messages:read`** — Messages marked as read
  - Payload: Array of read status objects

- **`chat:updated`** — Chat settings changed
  - Payload: Updated chat object (includes `aiAgentEnabled`, `aiAgentVisibility`)

- **`chat:error`** — Error occurred during event handling
  - Payload: `{ event: string, message: string }`

## Key Business Logic

### Edit Window
Messages can only be edited or deleted within **5 minutes** of creation. This is enforced in `chat.service.ts`:
```typescript
const EDIT_WINDOW_MS = 5 * 60 * 1000;
```

### Auto-Generated Messages
Messages with `autoGenerated: true` cannot be edited or deleted. These are system messages typically created during document operations. Attempting to edit/delete throws `ForbiddenError: "Cannot edit/delete auto-generated messages"`.

### AI Agent Messages
The AI agent is **not** a `User` / `ChatMember`. Its messages have `isFromAiAgent: true` and `authorId: null`.

- **Per-chat toggle**: `Chat.aiAgentEnabled` (default `true`) controls whether the AI agent can post in a chat.
- **Visibility mode**: `Chat.aiAgentVisibility` is either `ALL` (default — every active member sees the AI reply) or `SENDER_ONLY` (only the member who triggered the AI sees the reply, enforced via `ChatMessage.visibleToMemberId`).
- **Visibility filter**: The last-message preview and unread counts always apply `visibleToMemberId IS NULL OR visibleToMember.userId = currentUser.id`. `getMessages` applies that same filter **only when the chat's `aiAgentVisibility = SENDER_ONLY`**; when `aiAgentVisibility = ALL`, all messages in the chat are returned regardless of `visibleToMemberId`.
- AI agent messages **cannot be edited or deleted** (throws `ForbiddenError`).

### Member Authorization
All message operations validate that the user is an active member of the chat:
- Member status must be `ChatMemberStatus.ACTIVE`
- For edit/delete: User must be the message author
- Checked via `chatMemberRepository`

### Read Status Tracking
- Tracked in `ChatMessageReadStatus` table with composite key: `(chatMessageId, readById)`
- `skipDuplicates: true` prevents duplicate read records
- Includes timestamp of when message was read

### Message Threading
Messages support threaded replies via `parentMessageId`:
- Optional field on message creation
- Queries can filter by `parentMessageId` to fetch thread

## Important Notes

1. **Cursor Pagination** — All list endpoints use cursor-based pagination, not offset-based
2. **Team Scoped** — All repositories are initialized with `teamId` to provide data isolation
3. **Transaction Context** — Operations using `withRepositories` run in transaction context
4. **Real-time Updates** — Socket.IO broadcasts to all users in chat room after operations
5. **No Soft Deletes** — Messages are hard-deleted, not soft-deleted

## AI Assistant Sub-module (`llm/`)

The `llm/` subfolder adds a **multi-agent AI assistant** that replies inside the same chat. Triggered when `sendMessage` is called with `isForAi: true` (REST body or `chat:message:send` socket payload) AND the chat has `aiAgentEnabled = true`.

### Privacy model
- The user's question is persisted with `visibleToMemberId = <senderChatMember.id>` — only the sender sees it.
- The AI reply is also persisted with `visibleToMemberId = <senderChatMember.id>`, `isFromAiAgent = true`, `authorId = null`.
- Both `chat.handler.ts` and `chat.socket-handler.ts` check `chatMessage.visibleToMemberId`: if non-null, the broadcast goes to `user:${userId}` instead of `chat:${chatId}`. AI replies are emitted from `aiMessagePersistenceService` directly to `user:${initiatorUserId}`.
- The existing `buildVisibilityFilter` in `chat.service.ts` already filters `getMessages` by `visibleToMemberId`, so privacy in message listings works automatically.

### Multi-agent architecture
`llm/graph/ai-graph.service.ts` runs a supervisor pattern:
1. **Router** — Gemini structured-output call picks 1..N specialized agents from `{members, documents, teamStats, chat}`.
2. **Sub-agents** — each is a `createReactAgent` (from `@langchain/langgraph/prebuilt`) with a small set of Zod-typed tools and an agent-specific prompt.
3. **Aggregator** — final Gemini call with structured output `{ content: string }`, validated via Zod; falls back to a safe error string on validation failure.

All read-only. Tools never see Prisma — they call **data services** that the model is forced to use.

### Data services (read-only, all in `llm/data/`)
- `chatDataService` (tenant DB) — `getChatScope`, `getChatInfo`, `getChatMembers`, `getHistoryForAi`, `findChatMemberByUserId`
- `membersDataService` (master DB) — `getTeamMembers`, `findMembersByName`, `getMemberByUserId`
- `documentsDataService` (tenant DB) — `listDocuments`, `getDocumentById`; both honor `scopedDocumentId` so a chat tied to one document cannot leak others
- `teamStatsDataService` (master DB) — `getTeamStats`, `getMemberCountsByRole`

### Awilix names
`llmService`, `aiGraphService`, `geminiProviderService`, `chatDataService`, `membersDataService`, `documentsDataService`, `teamStatsDataService`, `aiMessagePersistenceService`. All registered via `addDIResolverName(createService, "<name>")` at the bottom of each `*.service.ts` file (auto-loaded by `plugins/awilix.ts`).

### History fed to the AI
`chatDataService.getHistoryForAi` pulls the last 20 messages where `visibleToMemberId = userChatMemberId` AND (the user wrote it OR `isFromAiAgent = true`). Other members' messages are never mixed in.

### Required env
`GEMINI_API_KEY` (added to `src/plugins/env.ts` and `EnvConfig`).

### Out of scope right now
Streaming replies, write/action tools, RAG over document contents, per-tool ACL, tests.
